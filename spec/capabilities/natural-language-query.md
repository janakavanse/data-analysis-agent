# Capability: Natural Language Query

## What & why

A user types a natural-language question about a previously uploaded dataset in a multi-turn chat session.
The agent translates the question to a SQL SELECT statement, executes it against the correct SQLite table
(resolved from session context — the user does not need to name the table every turn), and returns the
results as a formatted answer: a markdown table of rows plus a prose summary addressing the question.
Session continuity is maintained by the LangGraph `AsyncSqliteSaver` checkpointer keyed to `thread_id` —
prior turns remain in context so follow-up questions naturally refer back to the same dataset and prior
results. Serves the "NL question produces correct SQL and grounded answer" and "follow-up works without
re-specifying dataset" success criteria in `spec/product.md`.

## Acceptance criteria (EARS — these ARE the eval inputs)

- WHEN the user asks a natural-language question about an uploaded dataset the system SHALL translate it to a valid SQL SELECT statement, execute it against the correct `ds_<uuid>` table, and return an answer that includes the raw result rows (as a markdown table) and a prose summary that directly addresses the question.
- WHILE in a multi-turn session WHEN the user asks a follow-up question without naming a dataset the system SHALL scope the SQL to the same table referenced in the prior turn and return a correct answer without prompting the user to re-specify the dataset.
- WHEN the query returns zero rows the system SHALL return a response stating that no data matched the query rather than fabricating results.
- IF the natural-language question would require a mutating SQL statement (DROP, DELETE, UPDATE, INSERT, ALTER, TRUNCATE) THEN the system SHALL refuse to execute it and explain that only SELECT queries are permitted.
- IF the SQL generated by the agent fails to execute (syntax error, non-existent column) THEN the system SHALL catch the error, report what went wrong, and offer to try a corrected query.

## Tools & layers touched

- tool: `list_datasets`  (in-process @tool — retrieves available datasets + schemas from the `datasets` table)
- tool: `query_dataset`  (in-process @tool — executes a validated SELECT against a `ds_<uuid>` table)
- tool: `finish`  (in-process @tool — returns the formatted answer)
- layers: Memory (short-term via `AsyncSqliteSaver` checkpointer, `thread_id` keyed) — `harness/patterns/memory.md`
- layers: Guardrails (`on_tool_call` hook blocks any non-SELECT SQL before execution) — `harness/patterns/guardrails-and-hitl.md`
- layers: Context engineering (active dataset schema injected into system prompt each turn) — `harness/patterns/context-engineering.md`

## Evaluation

- outcome evaluation_steps:
  - Does the answer include a markdown table or structured list of query result rows?
  - Does the prose summary directly address the original question using only values from the query results (no invented data)?
  - When asked a follow-up question in a multi-turn thread, does the answer correctly reference the same dataset without the user re-specifying it?
  - When the query returns zero rows, does the answer clearly state no data matched rather than fabricating rows?
- expect_tools: [query_dataset, finish]
- forbid_tools: []

## Notes

- The `query_dataset` tool must validate that the SQL contains only `SELECT` (case-insensitive) before
  executing. Any other DML/DDL keyword at the statement root triggers an immediate refusal response —
  this is enforced deterministically in the tool, not just by the guardrail.
- The active dataset context (table_name, schema_json) is injected at the top of the system prompt each
  turn by the context-engineering layer so the model does not need to call `list_datasets` on every turn —
  only when the user switches to a different dataset or the context is absent.
- `list_datasets` is available for the model to call explicitly when the user refers to a different
  dataset by name or when no active dataset is set.
- Results are capped at 1000 rows by default to prevent context overflow; the tool returns a count note
  if results were truncated.
- Out of scope for this capability: cross-dataset joins, parameterised queries, query history export,
  scheduled queries.
