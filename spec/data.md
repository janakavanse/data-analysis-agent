# Data Model

## Storage Technology

SQLite (via SQLAlchemy ORM, synchronous). Chosen for simplicity ‚Äî this is a single-process, single-user local server with no concurrent write requirements. Dataset metadata and session state fit comfortably in SQLite. Raw CSV rows are never stored in the database.

Migrations managed by Alembic. Run `uv run alembic upgrade head` before first use and after each schema change.

---

## Entities

### Entity: Session

Represents one browser session. Created automatically when the browser loads the page. Groups datasets and queries under a single ephemeral identity.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | TEXT (UUID) | yes | Primary key; generated by server |
| `created_at` | TIMESTAMP (UTC) | yes | When the session was created |

**Table name:** `sessions`

---

### Entity: Dataset

Represents one uploaded CSV file within a session. Stores file metadata only ‚Äî not raw rows.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | TEXT (UUID) | yes | Primary key; generated by server |
| `session_id` | TEXT (UUID) | yes | Foreign key ‚Üí `sessions.id` |
| `filename` | TEXT | yes | Sanitised original filename (e.g. `sales_2024.csv`) |
| `row_count` | INTEGER | yes | Number of data rows (excluding header) |
| `column_names` | TEXT (JSON array) | yes | Column names as a JSON array string; e.g. `["Month","Revenue","Region"]` |
| `file_path` | TEXT | yes | Absolute path to the file on disk; e.g. `data/uploads/<session_id>/<dataset_id>_<filename>` |
| `created_at` | TIMESTAMP (UTC) | yes | When the dataset was uploaded |

**Table name:** `datasets`

---

### Entity: Run

Represents one executed NL query. One run is created per `POST /sessions/{id}/queries` call. Extends the existing `RunRow` with analyst-specific fields.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | TEXT (UUID) | yes | Primary key; generated by server (existing column) |
| `status` | TEXT | yes | `"pending"` ‚Üí `"completed"` or `"failed"` (existing column) |
| `session_id` | TEXT (UUID) | yes | Foreign key ‚Üí `sessions.id` (new column) |
| `dataset_id` | TEXT (UUID) | no | Foreign key ‚Üí `datasets.id`; null when `dataset_ids` (multi-file) is used (new column) |
| `question` | TEXT | yes | The user's natural-language question (new column) |
| `answer_text` | TEXT | no | Gemini's prose answer; null on failure (new column, replaces `output_text` semantically) |
| `table_json` | TEXT (JSON array) | no | Serialised `table_data` as a JSON string; null if no table (new column) |
| `chart_b64` | TEXT | no | Base64-encoded PNG string; null if no chart (Phase 2 column) |
| `error_message` | TEXT | no | Error string on failure; null on success (existing column) |
| `input_text` | TEXT | no | Kept for backward compatibility; set to the question value (existing column) |
| `output_text` | TEXT | no | Kept for backward compatibility; set to the answer text value (existing column) |
| `created_at` | TIMESTAMP (UTC) | yes | When the run was created (existing column) |
| `updated_at` | TIMESTAMP (UTC) | yes | Last update time (existing column) |

**Table name:** `runs`

> **Note:** The existing `RunRow` model is extended in-place with the new columns. A single Alembic migration adds `session_id`, `dataset_id`, `question`, `answer_text`, `table_json`, and `chart_b64` (Phase 2) columns to the existing `runs` table.

## Relationships

```
Session (1) ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ< Dataset (many)
Session (1) ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ< Run (many)
Dataset (0..1) ‚îÄ‚îÄ< Run (many)    ‚Üê nullable: multi-file runs use dataset_ids (not stored as FK)
```

- A `Session` has zero or more `Dataset` records.
- A `Session` has zero or more `Run` records.
- Each `Run` optionally references one primary `Dataset` via `dataset_id` (Phase 1‚Äì2); multi-file runs (Phase 3) may have `dataset_id = null`.
- There is no explicit many-to-many join table for Phase 3 multi-file runs ‚Äî the `dataset_ids` list is a query-time input, not persisted (out of scope for Phase 1‚Äì3).

## Data Lifecycle

| Event | Effect |
|-------|--------|
| Browser page load | `POST /sessions` creates a `SessionRow` |
| CSV upload | `DatasetRow` created; file written to disk |
| NL query submitted | `RunRow` created with `status = "pending"` immediately; updated to `"completed"` or `"failed"` after graph execution |
| Server restart | In-memory state cleared; DB rows and uploaded files on disk persist across restarts |
| No explicit deletion | Sessions, datasets, and runs accumulate until the SQLite file or filesystem is cleared manually |

---

## Sensitive Data

- **Uploaded CSV files** may contain user data. They are stored on the local server filesystem only. They are never forwarded to external services (Gemini receives only schema + aggregate stats, not row contents).
- **No PII is explicitly collected.** Sessions are anonymous (no username or email).
- **API key (`AGENT_GEMINI_API_KEY`)** is stored in `.env` (gitignored) only; never written to the database or logged.
- No encryption at rest in Phase 1‚Äì3 (local-only deployment assumption).
